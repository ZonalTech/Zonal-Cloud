01 The core loop

03 The four app types

Each type is a "recipe": how to detect it, how to build it, and how it runs. Supporting these four cleanly is the heart of the platform.

① Static (React / Vite / Next export) simplest

  • Detect: vite/react-scripts/next in package.json; build script present.
  • Build: npm ci && npm run build → output dir (dist/build/out).
  • Run: serve static files (nginx container or proxy's file server). Cheap, scales trivially.

② Node.js backend (Express / Nest / API) long-running

  • Detect: a start script + server listening on a port.
  • Build: npm ci (+ optional build step for TS/Nest).
  • Run: container with injected PORT + env vars; health-checked; restart policy.

③ Full-stack (frontend + backend + DB) composed

  • Detect: monorepo or manifest (zonal.yaml) declaring services.
  • Build: build each service; provision a Postgres container per app.
  • Run: linked containers on a private network; DB URL injected as env.

④ Node-RED dev / flows

  • Detect: selected by user as app type (Node-RED is a fixed runtime image).
  • Build: base nodered/node-red image + user flows/settings/credentials.
  • Run: persistent volume for /data; editor secured behind auth; per-instance isolation.
Automatic builds via Nixpacks. For types ①–③, the user doesn't write a Dockerfile. Nixpacks (open-source, MIT) inspects the repo, detects the language/framework, generates a build plan, and produces a runnable image. The optional zonal.yaml overrides any guess (build cmd, output dir, env, services). Node-RED (④) is the exception — it runs from a fixed base image plus the user's flows, so it isn't Nixpacks-built.

3.4 Two surfaces: user dashboard vs admin panel

A multi-tenant platform needs two distinct frontends. The user dashboard is what every tenant sees — their own apps only. The admin panel is the operator console — it sees and controls everything, across all tenants. They share the API but are gated by role.

             ┌──────────────────────┐                 ┌────────────────────────────┐
  tenant ──▶ │   USER DASHBOARD     │                 │      ADMIN PANEL           │ ◀── operator
             │  role: user          │                 │  role: admin / superadmin  │   (you)
             ├──────────────────────┤                 ├────────────────────────────┤
             │ • my projects/apps   │                 │ • ALL users & tenants      │
             │ • deploy, logs, env  │                 │ • suspend / delete / reset  │
             │ • my usage           │                 │ • impersonate user          │
             │ • billing (self)     │                 │ • quotas & plans per tenant │
             └──────────┬───────────┘                 │ • ALL apps/deploys (stop)   │
                        │                              │ • platform metrics & health │
                        │        same API,             │ • audit log                 │
                        └──────────┐  role-gated  ┌────┤ • global settings           │
                                   ▼              ▼    └─────────────┬──────────────┘
                              ┌─────────────────────────────────────▼──────┐
                              │  API (NestJS) — RBAC middleware on every route│
                              │  user → own resources · admin → tenant scope  │
                              │  superadmin → everything + platform config    │
                              └───────────────────────────────────────────────┘

Admin panel — capabilities

  • Users: list/search, create, suspend/ban, delete, reset password, force-logout, change role.
  • Impersonate: "log in as" a user to debug their apps (audited).
  • Tenants / orgs: create orgs, move users, set plan tier.
  • Quotas: per-tenant caps — max apps, CPU/RAM/disk, build minutes, concurrent deploys.
  • All apps & deployments: view any tenant's apps; stop/restart/redeploy; kill runaway containers.
  • Platform health: node/container metrics, queue depth, build success rate, disk usage.
  • Audit log: who did what, when (deploys, role changes, impersonations).
  • Global settings: registry, Traefik, default build limits, signup open/closed, feature flags.

Roles & access (RBAC)

  • user — owns projects/apps in their org; sees only their own.
  • admin — manages users/apps within an org/tenant (team owner).
  • superadmin — full platform control + global settings; the operator (you).

Every API route runs through RBAC middleware. The admin panel is a separate React app (or a gated route tree) reachable only to admin/superadmin; the same token carries the role claim.

Safety: destructive admin actions (delete user, wipe app) require confirmation + are written to the immutable audit log. Impersonation sessions are time-boxed and flagged.

3.5 Auto-deploy via GitHub Actions

Push to your repo → GitHub Actions runs your workflow → it calls Zonal's deploy API with a scoped token → Zonal builds & deploys. The platform exposes a token-authenticated deploy endpoint; the rest is a tiny reusable workflow we ship.

# developer                  # GitHub Actions                 # Zonal Cloud
git push origin main  ──▶  .github/workflows/deploy.yml  ──▶  POST /v1/deploy
                            • on: push [main]                 Authorization: Bearer $ZONAL_TOKEN
                            • build/test (optional)           { app_id, ref, commit }
                            • call Zonal deploy API   ─────────────┐
                                                                   ▼
                                                         enqueue build job
                                                         build → image → run → route
                                                                   │
                            ◀──────── status + live URL ◀──────────┘
                            (deploy summary in the Actions run)

What Zonal provides

  • Deploy API: POST /v1/deploy (token auth, per-app scope).
  • Deploy tokens: created in dashboard, stored as a GitHub secret (ZONAL_TOKEN).
  • Reusable workflow / action: drop-in deploy.yml users copy into their repo.
  • Status callback: deploy result + live URL surfaced back in the Actions log.

Two trigger paths (both supported)

  • Push-driven (Actions): user's CI calls our API — works today, no DNS needed.
  • Webhook (later): GitHub App that auto-deploys on push without the user editing CI. Step 7.
Local CI testing with act. We test the GitHub Actions workflow locally using nektos/act (open-source) — it runs .github/workflows/deploy.yml inside Docker on this machine, no GitHub cloud and no tunnel needed. Since act runs on the same host as the platform, the workflow reaches the local deploy API directly (host network / host.docker.internal). Once the VPS is live, the real GitHub-hosted Actions use the public API URL — same workflow file, unchanged.

05 Tech stack (proposed)

Control plane

  • API: Node.js + NestJS (structure, DI, fits Node-everywhere)
  • DB: Postgres + Prisma
  • Queue: Redis + BullMQ
  • Auth: JWT sessions; OAuth (GitHub) later

Frontends & CLI

  • User dashboard: React + Vite + Tailwind
  • Admin panel: React (role-gated) — tenant & user control
  • Live logs: WebSockets / SSE
  • CLI: Node (zonal deploy, logs, env)

Build & runtime

  • Engine: Docker (dockerode lib)
  • Builder: Nixpacks MIT · automatic — auto-detects & builds, no Dockerfile
  • Proxy: Traefik MIT · self-hosted — label routing + ACME
  • Registry: Docker registry container
Verify during implementation: exact Traefik v3 label syntax, Nixpacks vs hand-written Dockerfiles for build, and Node-RED multi-tenant auth hardening. These are the spots to confirm against current docs when we reach them — the architecture doesn't depend on the specifics.

06 Security & isolation (non-negotiables)

  • Build isolation: each build in its own container; drop network after dependency install; CPU/memory/time limits.
  • Runtime isolation: per-app Docker network; no access to the Docker socket from user containers.
  • Socket exposure: only the builder/orchestrator touches docker.sock — never user code.
  • Resource quotas: per-app CPU/RAM/disk caps; restart limits to stop crash loops.
  • Secrets: never baked into images; injected at runtime; encrypted at rest in DB.
  • Node-RED: editor behind auth, projects sandboxed, function-node access restricted.
  • TLS everywhere on VPS; rate-limit the API; audit log of deploy actions.
